# AI Summarization — Test Quality Review

The suite has useful offline coverage across worker, Edge Function, frontend, and Playwright smoke paths, but several tests assert mocked outcomes rather than proving the production contract. Overall risk rating: **Medium-High** because core acceptance gates (full personal-name redaction, extracted-text 512 KB cap, exact workflow status ordering, Azure HTTP error behavior, and no-network-by-default for Deno tests) are not fully encoded.

## Testing Gaps

### Worker

- **[High] Untitled personal names are not tested against the SPEC's broader requirement.** Source: SPEC requires first names and surnames redacted before model/persistence (`docs/specs/ai-document-summarization.md:28-29`, `:67-69`, `:231-233`, `:337-338`); ADR-0004 repeats "Redact before sending" (`docs/adrs/0004-use-anthropic-claude-for-document-summarization.md:81-86`). Existing tests cover title-prefixed names (`temporal/tests/test_summarization.py:22-31`) and a non-person capitalized place (`:34-36`), while production intentionally only redacts title-prefixed names (`temporal/src/activities/summarization.py:27-37`). **Why it matters:** `Jean Dupont a signé` would reach Claude despite the acceptance gate saying names must not appear. **Suggested test:** add cases for `Jean Dupont a signé`, `Claire Bernard`, hyphenated/accented names, and non-name capitalized words; assert intended behavior explicitly.

- **[High] Workflow status transition order is not asserted.** Source: SPEC mandates `uploaded -> extracting -> summarizing -> completed` and failure `failed` (`docs/specs/ai-document-summarization.md:104-107`, `:217-230`); ADR-0003 requires status transitions tested (`docs/adrs/0003-orchestrate-ai-summarization-with-temporal-workflow.md:56-62`). `test_workflow_completes_and_persists_summary` only checks final row fields (`temporal/tests/test_workflow_integration.py:108-112`), and the fake store overwrites status (`:63-66`). **Why it matters:** the workflow could skip `extracting` or `summarizing` and still pass. **Suggested test:** capture every `update_row` call and assert exact status sequence plus `save_summary` final payload.

- **[High] Extracted/decoded text over 512 KB is untested.** Source: SPEC says extraction should reject decoded text exceeding 512 KB (`docs/specs/ai-document-summarization.md:217-222`) and resolved max input is 512 KB (`:370`). Tests cover only downloaded raw bytes over a tiny cap (`temporal/tests/test_ai_activities.py:59-65`); production checks `len(data)` before parsing (`temporal/src/activities/ai_summarization.py:32-38`) and never checks extracted text length. **Why it matters:** a compressed DOCX/PDF below 512 KB can expand to far more text and bypass the cap. **Suggested test:** generate or monkeypatch a DOCX/PDF extraction returning `MAX+1` bytes of text and expect `InputTooLarge`.

- **[High] Azure non-2xx, timeout, and retry behavior are not covered.** Source: SPEC requires bounded retries on `429`/`5xx`/timeouts and clean failed status (`docs/specs/ai-document-summarization.md:80-82`, `:234-241`). `test_summarize_claude.py` fakes `raise_for_status()` as a no-op (`temporal/tests/test_summarize_claude.py:15-24`) and only tests provider-not-configured (`:85-91`). **Why it matters:** 400/401/429/500/timeout handling could be wrong or leak raw errors. **Suggested test:** use fake responses raising `httpx.HTTPStatusError`/`TimeoutException`; for workflow, use an activity fake that fails transiently then succeeds and another that exhausts retries and verifies `mark_failed`.

- **[Medium] Malformed PDF/DOCX and scanned/image-only PDFs are undercovered.** Source: SPEC calls out malformed/invalid inputs and scanned PDFs should fail clearly (`docs/specs/ai-document-summarization.md:44`, `:57-59`, `:353-354`). Tests stub PDF parsing (`temporal/tests/test_extract_text.py:48-65`) and only check `None` page text at the pure helper level (`:68-81`); DOCX happy path uses a real generated document (`:21-32`) but no corrupt ZIP. **Why it matters:** parser exceptions may be retried as transient instead of non-retryable user errors. **Suggested test:** malformed PDF bytes and corrupt DOCX bytes through `ai_summarization.extract_text`, expecting a clean non-retryable error and no document content in the message.

- **[Medium] Supabase REST failure paths are missing.** Source: SPEC requires failures surface clean `failed` status and retry path (`docs/specs/ai-document-summarization.md:80-82`, `:229-230`). `test_supabase_rest.py` always returns successful fake responses (`temporal/tests/test_supabase_rest.py:16-26`). **Why it matters:** download/update/claim HTTP failures may not be classified or retried as intended. **Suggested test:** fake `raise_for_status()` failures for `download_object`, `update_row`, `fetch_pending`, and `claim_request`; assert retryable propagation or poller survival as appropriate.

- **[Medium] Map-reduce tests do not prove chunk fidelity.** Source: SPEC requires chunking above 128 KB and reduce behavior (`docs/specs/ai-document-summarization.md:226-227`, `:370-371`). Existing test forces a 40-byte threshold (`temporal/tests/test_summarize_claude.py:71-82`) and only asserts more than one call and a reduce prompt. **Why it matters:** chunks could exceed the threshold, omit text, duplicate text, or reduce the wrong content while the test still passes. **Suggested test:** record every payload; assert each map `messages[0].content` is within threshold and covers the input, and final reduce input equals the joined map responses.

- **[Low] Least-privilege egress and no-content logging are not encoded.** Source: SPEC forbids document content in logs and limits external I/O to `summarize_with_claude` (`docs/specs/ai-document-summarization.md:70-73`; ADR-0003 `:63-66`). Tests do not capture logs or forbid `httpx.Client` outside the Claude path. **Why it matters:** future changes could leak source text or add accidental external calls. **Suggested test:** caplog assertions for representative text/name/secret absence; monkeypatch `httpx.Client` to fail if used by extraction/status/save paths.

### Edge Function

- **[High] Default Deno tests may perform network I/O.** Source: SPEC says default tests make no network calls (`docs/specs/ai-document-summarization.md:78-79`, `:306-323`). `router.test.ts` imports assertions from `https://deno.land/std@0.224.0/assert/mod.ts` (`supabase/functions/summaries/router.test.ts:6-9`). **Why it matters:** first-run CI or uncached local runs may contact `deno.land`, violating offline-by-default. **Suggested test/setup change:** use vendored/local assertions, JSR lock/cache with `--cached-only`, or project-local helpers.

- **[High] `size_bytes` validation gaps are untested.** Source: Edge validation must enforce 512 KB server-side and reject invalid inputs (`docs/specs/ai-document-summarization.md:57-59`, `:196-212`; ADR-0005 `:72-76`). Tests cover only `MAX_INPUT_BYTES + 1` (`supabase/functions/summaries/router.test.ts:112-124`). Production treats missing `size_bytes` as `0` and accepts non-finite values because it only rejects finite values over the cap (`supabase/functions/summaries/router.ts:48`, `:57-62`). **Why it matters:** clients can omit or send `NaN`/negative sizes and still create requests. **Suggested test:** require integer `size_bytes`, reject missing/non-numeric/negative, accept exactly `MAX_INPUT_BYTES`.

- **[Medium] Insert/select/update contracts are not asserted.** Source: SPEC requires request row creation with storage path, original filename, MIME, status uploaded, workflow null (`docs/specs/ai-document-summarization.md:196-203`). The valid create test only checks status (`supabase/functions/summaries/router.test.ts:63-79`), not all inserted fields; retry test checks only `status` and `workflow_id` (`:167-178`). **Why it matters:** the router could omit `storage_path` or fail to clear `summary/error_message` and tests would pass. **Suggested test:** assert full insert/update payloads and selected response shape.

- **[Medium] RLS/ownership behavior is not tested.** Source: SPEC requires RLS owner access and private rows (`docs/specs/ai-document-summarization.md:62-64`, `:188-189`); ADR-0005 requires RLS-aware access (`docs/adrs/0005-supabase-edge-function-summarization-entry-point.md:77-82`). Router tests use a service-role fake and do not assert `created_by` or user scoping. **Why it matters:** list/get could expose summaries across users. **Suggested test:** add migration/policy integration tests or router tests that include authenticated user context and owner filters if the function is meant to scope results itself.

### Frontend

- **[High] API payload contract is not tested.** Source: SPEC defines `POST /summaries` payload including `storage_path`, `original_filename`, `mime_type`, and `size_bytes` (`docs/specs/ai-document-summarization.md:196-202`). Component sends those fields (`frontend/src/components/engine/ai/AiSummarizer.tsx:133-141`), but tests only assert POST URL/method (`frontend/src/components/engine/ai/AiSummarizer.test.tsx:99-102`). **Why it matters:** a broken body could still render the mocked summary. **Suggested test:** assert `JSON.parse(fetchMock.mock.calls[0][1].body)` exactly and assert Supabase upload path/content type/upsert options (`AiSummarizer.tsx:127-130`).

- **[Medium] Status UI coverage is incomplete.** Source: SPEC acceptance criteria require visible `uploaded`, `extracting`, `summarizing`, `completed`, `failed` (`docs/specs/ai-document-summarization.md:102-107`). Tests cover completed and failed summaries (`frontend/src/components/engine/ai/AiSummarizer.test.tsx:79-138`) but not intermediate status labels mapped in `AiSummarizer.tsx:48-55`. **Why it matters:** progress UX can regress while tests pass. **Suggested test:** return each status from the poll endpoint and assert the visible badge/disabled controls.

- **[Medium] Retry behavior is only partially covered.** Source: retry path is required (`docs/specs/ai-document-summarization.md:106-107`, `:207-209`). Tests verify a Retry button appears (`frontend/src/components/engine/ai/AiSummarizer.test.tsx:112-138`) but never click it; component silently ignores non-OK retry responses (`frontend/src/components/engine/ai/AiSummarizer.tsx:157-170`). **Why it matters:** retry could stop working or hide server errors. **Suggested test:** click Retry, assert `POST /summaries/{id}/retry`, uploaded status reset, polling resumes, and 409/500 shows a user-visible message.

- **[Medium] XSS-safe rendering is not asserted.** Source: SPEC forbids unsafe HTML insertion (`docs/specs/ai-document-summarization.md:65-66`). Component renders `{record.summary}` as text (`frontend/src/components/engine/ai/AiSummarizer.tsx:253-254`), which is good, but there is no regression test. **Why it matters:** a future refactor to raw HTML would be high risk. **Suggested test:** mocked summary `<img src=x onerror=alert(1)>` should appear as text and no `img` element should exist.

- **[Low] Test globals may leak between tests.** `src/test/setup.ts` clears mocks and cleans up DOM (`frontend/src/test/setup.ts:10-13`) but does not call `vi.unstubAllGlobals()`. Flow tests stub `fetch` (`frontend/src/components/engine/ai/AiSummarizer.test.tsx:89`, `:121`). **Why it matters:** later tests may accidentally use stale mocked `fetch`. **Suggested test/setup:** enable `unstubGlobals` in Vitest or call `vi.unstubAllGlobals()` in `afterEach`.

### E2E

- **[Medium] Offline smoke tests do not validate real contracts.** Source: SPEC requires local-stack integration for upload and summary persistence (`docs/specs/ai-document-summarization.md:319-322`). Smoke routes mock Storage and Edge responses (`frontend/e2e/ai-summarize.smoke.spec.ts:14-48`, `:73-105`) and never inspect request payloads. **Why it matters:** route, payload, or storage contract regressions can pass. **Suggested test:** in route handlers, parse POST body and upload request; assert expected path, MIME, size, auth headers, and no unexpected real backend requests.

- **[Medium] Live redaction/English assertions are too weak.** Source: SPEC requires English summary and no personal names (`docs/specs/ai-document-summarization.md:25-29`, `:337-338`). Live E2E only asserts length and absence of `Dupont` (`frontend/e2e/ai-summarize.live.spec.ts:53-59`), not absence of `Jean`, `Monsieur`, or that output is English. **Why it matters:** a French output or partial name leak can pass. **Suggested test:** assert no `/Jean|Dupont|Monsieur/i`, assert presence of English cues from the fixture, and assert absence of French-only phrases from the source.

## Tests That May Pass For The Wrong Reasons

- **`temporal/tests/test_workflow_integration.py::test_workflow_completes_and_persists_summary` (`:96-112`)** — The mocked Azure client always returns `"An English summary."` (`:36-55`), and the test asserts that exact mocked value was persisted (`:111`). It would pass if the workflow skipped `redact_personal_names`, sent the unredacted `"Monsieur Jean Dupont"` fixture (`:69-70`) to Claude, or skipped intermediate statuses. **Harden by** recording the Claude request payload and Supabase update history; assert redacted text reaches `_call_claude`, names are absent, and statuses occur in order.

- **`temporal/tests/test_workflow_integration.py::test_workflow_marks_failed_on_unsupported_mime` (`:115-130`)** — It proves one non-retryable extraction error reaches `failed`, but not that `mark_failed` is called after summarization/provider/persistence failures or that the error is clean/non-leaky. **Harden by** adding separate workflow tests where `summarize_with_claude` raises `ProviderNotConfigured`/HTTP errors and `save_summary` fails.

- **`temporal/tests/test_ai_activities.py::test_redact_personal_names_activity` (`:86-89`)** — It only asserts `"Jean Dupont"` is absent. A broken implementation returning `""`, replacing the whole sentence, or removing non-name text would pass. **Harden by** asserting placeholder presence and preservation of `"Le rapport"` / `"est complet"`, plus untitled-name behavior.

- **`temporal/tests/test_summarize_claude.py::test_summarize_direct_path` (`:58-68`)** — The result assertion is the fake response body (`:30`, `:43`), and the fake never enforces Azure's real response/status contract. It does not assert the URL path/API version, `anthropic-version`, `content-type`, `messages` content, or system prompt. **Harden by** asserting the complete request shape and adding malformed response/no text blocks tests.

- **`temporal/tests/test_summarize_claude.py::test_summarize_chunks_large_input_with_map_reduce` (`:71-82`)** — The fake returns the same text for every map and reduce call, so `result == "An English summary."` (`:77`) says little about production logic. `len(posts) > 1` and any reduce prompt (`:78-82`) would pass even if chunks lost content or exceeded limits. **Harden by** making fake responses depend on input (`summary-of-chunk-N`) and asserting final reduce content is the joined partials.

- **`temporal/tests/test_extract_text.py::test_extract_pdf_joins_pages` (`:48-65`)** — The test title suggests PDF extraction, but `pypdf.PdfReader` is fully faked (`:58-63`), so it only tests page joining. A real `pypdf` integration failure or malformed PDF behavior would not be caught. **Harden by** adding a small real text PDF fixture/generated PDF test and corrupt PDF error test.

- **`temporal/tests/test_poller_integration.py::test_poll_once_claims_then_starts_workflow` and `::test_poll_once_processes_each_pending_row` (`:33-49`, `:62-70`)** — The fake `claim_request` ignores its arguments (`:35`, `:64`). If `_poll_once` computed the wrong workflow id for the claim, the tests could still pass because they only inspect `start_workflow`. **Harden by** recording `claim_request` calls and asserting `(request_id, workflow_id)` for each row.

- **`supabase/functions/summaries/router.test.ts::POST create: valid payload returns 201...` (`:63-79`)** — The fake builder ignores `.select()` and `.single()` arguments (`:37-41`), and the test only checks inserted `status` (`:77-78`). It would pass if `storage_path`, `original_filename`, or `mime_type` were omitted from the insert. **Harden by** asserting the entire insert payload and returned JSON shape.

- **`supabase/functions/summaries/router.test.ts::POST retry: failed request returns 200...` (`:167-178`)** — The fake ignores `.eq("id", id)` (`:37-38`), so the test would pass if production updated every failed row instead of the target id. It also does not assert `summary` and `error_message` are cleared even though production does (`supabase/functions/summaries/router.ts:120-130`). **Harden by** capturing method arguments (`eq`, `select`, `order`, `limit`) and asserting full update payload.

- **`frontend/src/components/engine/ai/AiSummarizer.test.tsx::uploads, creates a request...` (`:79-108`)** — The mocked fetch returns a completed summary regardless of request body (`:82-88`), and assertions only check URL/method (`:99-102`) plus rendered mock text (`:103-107`). It would pass if the component sent the wrong `storage_path`, omitted `size_bytes`, or used wrong auth headers. **Harden by** asserting upload args and POST body/headers before returning the mock response.

- **`frontend/src/components/engine/ai/AiSummarizer.test.tsx::shows an error and a Retry button...` (`:112-138`)** — It never clicks Retry, so it does not exercise `onRetry` (`frontend/src/components/engine/ai/AiSummarizer.tsx:157-170`). A broken retry endpoint path or missing poll restart would pass. **Harden by** clicking Retry and asserting the retry POST and follow-up polling.

- **`frontend/e2e/ai-summarize.smoke.spec.ts::uploads a document...` (`:13-70`)** — The route mock returns success for any POST ending in `/summaries` and never checks body/headers (`:24-48`). The test mainly proves the UI can render a mocked summary, not that it honors the Edge Function contract. **Harden by** route-level assertions on request payload and Storage upload request.

- **`frontend/e2e/ai-summarize.live.spec.ts::summarizes a real French document...` (`:30-59`)** — The redaction assertion checks only `/Dupont/i` (`:58`). A summary containing `Jean` or `Monsieur` would pass, and there is no strong English-output check. **Harden by** asserting all source personal-name tokens are absent and source French phrases are not reproduced.

## Mock/Fake Fidelity Risks

- **Deno Supabase fake is too permissive.** `makeFakeDb` returns one shared builder and ignores all query-builder arguments (`supabase/functions/summaries/router.test.ts:27-42`). It cannot catch missing `eq` filters, wrong selected columns, missing ordering, missing limit, or unscoped retry updates.

- **Azure/httpx fakes do not model error or response contracts.** `_FakeResponse.raise_for_status()` is always a no-op and `.json()` always returns a happy Anthropic-like content block (`temporal/tests/test_summarize_claude.py:15-24`; `test_workflow_integration.py:36-42`). Real Azure may return non-2xx, timeout, error JSON, empty content, multiple text blocks, or provider-specific headers; none are exercised.

- **Workflow integration fake Supabase store loses history.** The fake `update_row` overwrites fields in a dict (`temporal/tests/test_workflow_integration.py:61-66`), so it cannot verify transition ordering or whether a transient failed state was written before completion.

- **Poller fake Temporal client cannot expose idempotency/race issues.** `_FakeClient.start_workflow` only records calls (`temporal/tests/test_poller_integration.py:16-22`) and never raises `WorkflowAlreadyStartedError` or validates task queue/workflow id uniqueness.

- **Frontend fetch/storage mocks ignore production payload shape.** Component tests mock `supabase.storage.from().upload` with only `{ error: null }` (`frontend/src/components/engine/ai/AiSummarizer.test.tsx:14-22`, `:36-39`) and mocked `fetch` responses (`:82-89`, `:115-121`) without verifying request body, auth headers, or response edge cases.

- **Playwright smoke routes may mask real network leakage.** The `**/summaries**` and `**/storage/v1/object/**` handlers fulfill known calls but fall through with `route.continue()` for unexpected summaries traffic (`frontend/e2e/ai-summarize.smoke.spec.ts:47`, `:104`). Without a catch-all fail-fast route for backend/Azure URLs, accidental network calls could slip through.

## Recommendations

1. **High — Add redaction contract tests for untitled names and preserve/non-name cases.** Decide whether title-only redaction is acceptable; if not, tests should fail until implementation meets the SPEC.
2. **High — Record and assert workflow update history.** Verify exact `extracting -> summarizing -> completed` sequence and failure `mark_failed` paths for extraction, provider, and persistence failures.
3. **High — Enforce and test 512 KB consistently.** Add exact-boundary, missing/invalid `size_bytes`, and post-extraction decoded-text cap tests.
4. **High — Replace remote Deno test imports or enforce cached/offline Deno runs.** This is necessary to satisfy "no network in default tests."
5. **Medium — Harden Azure tests.** Assert full request shape, prompt content, chunk payload sizes/coverage, response parsing edge cases, 429/5xx/timeouts, and provider-not-configured through the workflow.
6. **Medium — Make Edge Function fake contract-aware.** Capture `select`, `eq`, `order`, `limit`, `insert`, and `update` arguments; assert full create/retry/list/get contracts.
7. **Medium — Strengthen frontend tests.** Assert upload options, POST body/headers, intermediate statuses, retry click flow, copy behavior, and XSS-safe rendering.
8. **Medium — Add fail-fast network guards in default tests.** In Vitest/Playwright, reject unexpected `fetch`/backend/Azure calls; in Python, consider socket/httpx guards except where explicitly mocked.
9. **Low — Add log-safety tests.** Capture logs from worker failure paths and assert no document text, summaries, names, or secrets appear.
